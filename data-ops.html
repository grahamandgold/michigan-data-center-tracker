Keeping the Tracker Live: Daily Data Operations & Agent Plan

Everything that has to be watched, verified, and refreshed to keep the Tracker current — and how to hand that work to a small team of automated agents instead of doing it by hand.

Source of truth: map-data.json83 sourced points · 6 layersNon-negotiable rule: every record cites a source

01The model, in one paragraph

The whole site is drawn from a handful of version-controlled JSON files. map-data.json holds 83 point records across six layers (projects, moratoria, generation, transmission, meetings, policy); live-data.json holds stories + meetings; water-live.json holds live Great Lakes levels; geo/*.geojson holds boundaries and the grid. The homepage counts, the map pins, the ticker, and every project page are computed from those files — no number is typed by hand. So "updating the site" is really "keeping those files fresh and correct." Every plotted record carries source_url, source_name, verified_date, and confidence. That sourcing discipline is the product; automation must preserve it, never shortcut it.

03Trust rules every check must obey

  • Every point cites a source. No source_url, no publish — the record goes to a "needs source" holding list, not the map.
  • Confidence stays visible. "Reported" is never silently promoted to "Confirmed"; promotion requires a second independent source.
  • Blanks stay blank. Undisclosed load renders "Undisclosed," never 0. Don't infer numbers.
  • Stats are computed, not typed. Counts and disclosed-load totals are recomputed from map_points on every write; humans never edit the ribbon numbers.
  • Stamp freshness. Every touched record gets a new verified_date; updated_at at the top of the file bumps on any change.

Why this split worksEach agent's sources, schema, and failure modes are narrow enough to reason about and to source-check. The orchestrator is the only thing that writes the published files, so validation and the "everything is sourced" rule are enforced in exactly one place — not seven.

05Where this should live (not your desktop)

The desktop is the single riskiest place for the source of truth: no history, no backup, no audit trail, and the site can't deploy from it. The fix is to make the Git repo the canonical store and the deploy target at once — the site already publishes from GitHub Pages at /mi-data-center-tracker/, so the data belongs in that same repo.

Recommended setup

  • Canonical store → the GitHub repo. map-data.json, live-data.json, water-live.json, and geo/ live in-repo. Git history becomes the audit trail: who changed which point, when, and citing what. That history is the credibility model.
  • Agents → GitHub Actions (cron). Tier 2–4 runs are scheduled Actions that execute the agents and open a PR. No server to babysit; the automation lives beside the code it edits.
  • High-frequency Tier 1 → a tiny always-on worker. Hourly wire + water levels are better on a small always-on host (Fly.io / Railway / a $5 VPS, or Val Town for the simplest cron) that commits water-live.json and queues stories — Actions cron granularity is coarse for hourly work.
  • Staging vs. live. Agents write to data/staging/*.json; the orchestrator validates and opens a PR; on merge, the published data/*.json updates and Pages redeploys. Nothing reaches readers without passing validation and the human gate.
  • Secrets & backup. Any model/API keys go in repo secrets (sources are keyless today). The repo history is the backup; optionally mirror map-data.json nightly to cheap object storage (R2/S3).
The one rule for hostingThe repo is the source of truth and the thing that deploys. Agents propose via pull request, you approve, Pages publishes. Nothing the site depends on should exist only on a laptop.

06First build for Claude Code

  1. Move data into the repo under data/ with a staging/ + published split, a JSON Schema for the point model, and a npm run validate that fails on a missing source_url, a bad confidence, or a hand-typed stat.
  2. Add id/slug + sources[] to project records — needed for stable project permalinks and multi-citation detail pages.
  3. Scaffold the Orchestrator as a Node runner: execute agent modules → validate → dedupe → recompute stats → write staging → open a PR (wired as a GitHub Action).
  4. Ship A6 Wire + A5 Water first — highest frequency, lowest risk, keyless sources — to prove the propose-validate-approve-publish loop end to end.
  5. Add a review view (PR template or a /review diff page) that shows each proposed change beside its source link, so the human gate is a two-minute approve.
